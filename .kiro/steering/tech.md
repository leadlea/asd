# Technical Stack

## プロジェクトの性格
このワークスペースは、Lead lea / 福原玄が扱う研究・実装系プロジェクトを前提とする。
特に以下の用途を想定する。
- NCNP共同研究
- 会話特徴量分析
- 論文化支援
- AWS / Bedrock を前提としたAI実装
- 可視化、提案資料化、Runbook化

## 優先技術スタック
- Python 3.12 を優先
- データ処理は pandas / numpy を優先
- 機械学習は scikit-learn を優先
- 可視化は matplotlib を基本とし、必要に応じて seaborn や plotly を検討
- ドキュメントは Markdown を基本とする
- 提案資料や共有資料は HTML + CSS + mermaid を好む
- AWS 実装は Bedrock / Lambda / API Gateway / S3 / DynamoDB / CloudFront / CloudWatch / IAM を優先的に想定
- IaC や運用再現性を重視する

## コーディング原則
- 既存ロジックを不用意に壊さない
- まず現状把握し、差分最小で変更する
- 変更時は目的と影響範囲を明確にする
- 再現コマンドや実行例を残す
- 一時的な実験コードと恒久コードを分ける
- パスや定数はハードコードしすぎない
- ログ、例外処理、前提条件チェックを意識する

## 研究コード原則
- 乱数シードを明示する
- 入出力ファイルの所在を明確にする
- 元データ、加工データ、成果物を分離する
- 統計評価や集計条件はコードと文書の両方に残す
- notebooksだけに依存せず、必要なら再現スクリプトを用意する

## ディレクトリ運用の考え方
- docs/ : 共有用の文書、Runbook、作業ログ
- scripts/ or src/ : 再現可能な処理コード
- artifacts/ : 生成成果物
- data/ : 入力データや加工データ
- paper/ : 論文原稿、図表、補足資料
- .kiro/steering/ : Kiroに与える判断基準と文脈

## Kiroへの期待
- 既存スタックに寄せて提案する
- 新技術を出すときは、なぜ必要かを説明する
- AWS実装ではセキュリティ、コスト、運用性も考慮する
- 研究支援では再現性と査読耐性を優先する
- ドキュメント出力では、そのまま共有できる完成度を目指す
