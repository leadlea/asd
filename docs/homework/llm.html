提案で頻出する用語集 ＋ LLMOps / Amazon Bedrock カスタムモデル整理（2026年2月版）

目的：提案先（例：芙蓉リース様）で混乱しやすい用語を短時間で共通言語化し、あわせて LLMOpsの範囲とAmazon Bedrockで「カスタム可能なモデル」を、運用・監査まで含めて整理します。
作成：国立機関連携AI研究者 / ビジョン・コンサルティング CAIO　福原 玄

0. 先に結論（誤解が起きやすいポイント）

  • LLMOpsは「強化学習やファインチューニング」だけではありません。評価（Evals）・安全（ガードレール）・監視（モニタリング）・変更管理（版管理）まで含めた運用の型です。
  • Amazon Bedrockで“オリジナルモデル”を作るとは、一般に「(A) RAG/プロンプト/ツール連携で挙動を制御」＋「(B) 必要ならカスタマイズ（SFT/継続事前学習/蒸留/RFT/インポート）」の組み合わせを指します。
  • ガードレール（Guardrails）は“モデルそのもの”ではなく、入力/出力に対する政策レイヤです。基盤モデルでも、カスタムモデルでも（対応APIの範囲で）同様に適用できます。

実務上のおすすめ：まずRAG＋Evals＋ガードレールで「安全・再現・説明」を固め、
その後、精度・コスト・応答速度・文体統一の要件が明確になった段階でモデルカスタマイズを検討します。

1. 提案で頻出する用語（最短の共通言語）

ICT

2. LLMOpsとは（“強化学習/ファインチューニング”を含むが、それだけではない）

LLMOpsは、LLMを業務で止めずに、安全に、継続改善するための運用工学です。
具体的には、Prompt/RAG、評価（Evals）、安全（Guardrails）、監視（モニタリング）、版管理（CI/CD）、監査ログまでを一体設計します。

flowchart TD A["要件・SLA・禁止事項
（業務/法務/セキュリティ）"] --> B["仕様化（Spec）
プロンプト/ポリシー/入出力形式"] B --> C["RAG/ツール連携/ワークフロー
(業務データ + ルール)"] C --> D["Evals（自動評価）
品質ゲート/回帰テスト"] D --> E["Guardrails（安全）
禁止話題/PII/攻撃耐性"] E --> F["本番運用（Observability）
遅延/失敗率/禁止回答率/コスト"] F --> G["改善サイクル（版管理）
Prompt/RAG/モデルの更新"] G --> D H["モデルカスタマイズ（必要時）
SFT/継続事前学習/蒸留/RFT/インポート"] -.-> G

整理：強化学習（RFT）やファインチューニング（SFT）は「改善手段の一部」です。
それ以前に、評価（Evals）と安全（Guardrails）の“運用ゲート”がないと、改善しても品質が担保できません。

3. Bedrockで「オリジナルモデル」を作れますか？

はい。ただし実務上は、いきなり学習ではなく、次の順で“勝ち筋”を作るのが堅実です。

  • Step 1 RAG＋プロンプト標準化：社内知識・規程・FAQを参照して回答の根拠を固定
  • Step 2 ガードレール：禁止領域・PII・攻撃（プロンプトインジェクション）対策
  • Step 3 Evals：定量評価で品質ゲートを作る（回帰で壊れない）
  • Step 4 必要ならモデルカスタマイズ：SFT/RFT/蒸留/インポート等

ガードレールは「モデルに混ぜ込む」より、外付けの政策レイヤとして運用する方が、更新・監査・適用範囲の管理が容易です。

C「RFT（強化学習）」対応（Bedrock公式の整理）

BedrockのRFTは、報酬関数（例：Lambda）で点数化しながらモデルを改善する方式です。
公式ページ上の「Supported models」は Amazon Nova Lite 2.0 が対象として記載されています。

RFTは、教師ありの正解データが作りにくい（しかし良し悪しは評価できる）領域で有効です。
実装はSageMaker Training Job（専用コンテナ）を用いる形で案内されています（下のハンズオン参照）。

D「カスタムモデル・インポート（Custom Model Import）」

Bedrockでは、対応アーキテクチャのオープンモデル等をインポートして、Bedrock経由で推論に利用できます。
公式情報では、インポート機能の対応リージョンとして eu-central-1 / us-east-1 / us-east-2 / us-west-2 が示されています。

参考情報（公式の注意点の例）：gpt-oss-120b / gpt-oss-20b のインポートは us-east-1 の記載、
また Qwen3 は「Converse API未対応」等の制約が記載されています。
（提案時は「どのAPI/機能が使えるか」を前提に、運用設計（ガードレール適用可否等）を詰めるのが安全です。）

6. Guardrails（ガードレール）は何を守るか（提案での言い方）

Guardrailsは、生成AIアプリの入力/出力に対して、禁止事項やセンシティブ情報の扱いをポリシーとして強制する仕組みです。
典型的には次を組み合わせます：

  • Content filters：有害カテゴリ（暴力/差別/性的/侮辱等）の強度設定
  • Denied topics / Word filters：禁止話題・禁止語のブロック
  • Sensitive info filters：PII等のマスキング/遮断（正規表現含む）
  • Contextual grounding：根拠参照（RAG）の整合性チェック（必要に応じて）
  • Automated reasoning policy：組織ポリシーの適用（要件に応じて）

提案上のキーメッセージ：「ガードレールは“モデル品質”ではなく“運用品質”」です。
監査・説明責任・事故防止（PII漏えい/禁止回答）を、仕組みで担保します。

7. ハンズオン（確認コマンド＋最小実装テンプレ）

7-1. まず“今のアカウント/リージョンで何ができるか”を確認

Bedrockはリージョンで機能差が出るため、最初に一覧化します（AWS CLI）。

# 1) CLIバージョン確認
aws --version

# 2) Fine-tuning対応モデル（リージョンごとに確認）
# 例：us-east-1
aws bedrock list-foundation-models \
  --by-customization-type FINE_TUNING \
  --region us-east-1 \
  --output table

# 例：us-west-2
aws bedrock list-foundation-models \
  --by-customization-type FINE_TUNING \
  --region us-west-2 \
  --output table

# 3) “モデルごとの対応（customizationsSupported）”を確認（JSONで見る）
aws bedrock list-foundation-models \
  --region us-east-1 \
  --output json \
  --query 'modelSummaries[?length(customizationsSupported)>`0`].[modelId,modelName,customizationsSupported]'

7-2. Guardrailを作成（最小例）

まずは「禁止話題」と「ブロック時メッセージ」を設定する最小構成例です。

# guardrail_min.json を作成（例）
cat << 'JSON' > guardrail_min.json
{
  "name": "proposal-guardrail",
  "description": "提案デモ用の最小ガードレール（禁止話題＋ブロック文言）",
  "topicPolicyConfig": {
    "topicsConfig": [
      {
        "name": "PII_Leakage",
        "definition": "個人情報（氏名・住所・電話・メール・口座等）の出力や推測、または入力からの再出力を要求する話題",
        "examples": [
          "この文章に含まれる電話番号を全部抜き出して",
          "顧客の住所と氏名を一覧で出して"
        ],
        "type": "DENY",
        "inputAction": "BLOCK",
        "outputAction": "BLOCK",
        "inputEnabled": true,
        "outputEnabled": true
      }
    ],
    "tierConfig": { "tierName": "CLASSIC" }
  },
  "blockedInputMessaging": "申し訳ありません。個人情報や機微情報に関わるため、その内容には対応できません。",
  "blockedOutputsMessaging": "申し訳ありません。安全上の理由により、その内容は出力できません。"
}
JSON

# 作成（例：us-east-1）
aws bedrock create-guardrail \
  --cli-input-json file://guardrail_min.json \
  --region us-east-1

# 返却された guardrailId を控える（以降のConverseで利用）
# 例：guardrailId = abcdef123456

7-3. Guardrail付きでConverse（推論）を実行

Guardrailの適用は、Bedrock RuntimeのConverseで指定します（traceを有効にすると検知内容が見えます）。

# messages.json を作成（例）
cat << 'JSON' > messages.json
[
  {
    "role": "user",
    "content": [
      { "text": "この文章から電話番号を抽出して一覧化して： 090-1234-5678" }
    ]
  }
]
JSON

# Guardrailを指定してConverse（例：Nova Lite）
# ※ modelId は利用可能なモデルに合わせて変更してください
aws bedrock-runtime converse \
  --region us-east-1 \
  --model-id "amazon.nova-lite-v1:0:300k" \
  --messages file://messages.json \
  --guardrail-config "guardrailIdentifier=abcdef123456,guardrailVersion=DRAFT,trace=enabled_full" \
  --output json

7-4. Bedrockで教師あり微調整（SFT）の“雛形”

公式のコードサンプルで案内されるように、学習データはJSONLで用意します（例は最小形）。
実務では「プロンプト/出力形式」「禁止事項」「根拠参照の有無」を揃えてから学習させます。 

# (例) 学習データ最小：train.jsonl（実際のスキーマはモデルごとに要確認）
cat << 'JSONL' > train.jsonl
{"input":"あなたは社内規程に厳密に従うアシスタントです。規程にないことは推測しません。","output":"了解しました。根拠が不明な場合は推測せず、必要な追加情報を確認します。"}
JSONL

# S3へ配置（例）
export REGION=us-east-1
export BUCKET="my-bedrock-customize-bucket-123456"
aws s3 mb "s3://${BUCKET}" --region "${REGION}"
aws s3 cp train.jsonl "s3://${BUCKET}/train/train.jsonl"

# カスタマイズジョブ作成（雛形）
# ※ 実運用では、データ形式・IAMロール・暗号化・評価データなどの要件を整備します。
cat << 'JSON' > sft_job.json
{
  "jobName": "proposal-sft-demo-001",
  "customModelName": "proposal-sft-model-001",
  "roleArn": "arn:aws:iam::123456789012:role/BedrockCustomizationRole",
  "baseModelIdentifier": "amazon.nova-lite-v1:0:300k",
  "trainingDataConfig": {
    "s3Uri": "s3://my-bedrock-customize-bucket-123456/train/"
  },
  "outputDataConfig": {
    "s3Uri": "s3://my-bedrock-customize-bucket-123456/out/"
  }
}
JSON

aws bedrock create-model-customization-job \
  --region "${REGION}" \
  --cli-input-json file://sft_job.json

# ジョブ状況確認
aws bedrock list-model-customization-jobs --region "${REGION}" --output table

7-5. RFT（強化学習）の“雛形”（SageMaker Training Job）

BedrockのRFTは、公式ページではSageMaker Training Job（専用コンテナ）で実行する流れが案内されています。
下は「方針を掴むための雛形」です（本番はネットワーク/IAM/ログ/コスト設計を合わせて実施します）。 

# 例：RFTの考え方（概要）
# - training data: OpenAI Reinforcement Fine-Tuning 形式（messages + reference_answer）
# - reward function: Lambda（評価してスコア返す）
# - training: SageMaker Training Job（公式ページにコンテナ例の記載あり）

# 実際の実行は、公式のテンプレート（ノートブック）に合わせるのが最短です。

運用上の注意：学習（SFT/RFT/インポート）は「できる/できない」だけでなく、
リージョン・API対応（Converse可否）・監査ログ・データ持ち出し要件まで含めて設計すると、提案の信頼性が上がります。
